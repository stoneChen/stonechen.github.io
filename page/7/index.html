聊一聊HTTP协议

这些天，开始全面进入项目开发了，卡在一个日历上，无奈自己写一个，回家还继续搞，所以就没写新文章。今天上班状态不错，把日历搞定了，还另外写了一个confirm组件，心情不错。

今天来聊聊HTTP协议。

长久以来对HTTP的认知只停留在GET,POST上，甚至对“协议”二字也模糊不清。直到这一两年从事前端开发，才对HTTP有了稍稍深入的了解。前几个礼拜看完了《图解HTTP》(日·上野宣 著)，这才对工作在web底层的HTTP协议有了一个较全面的认知。

先来讲讲“协议”吧，其实跟我们现实宏观世界类似，就是通信的双方遵循某种约定来一起完成某件事情。而HTTP(Hypertext Transfer Protocol )直译为超“文本”传输协议，本质上还是“文本”，即通讯时传输的只是字符串，通讯双方通过某些特殊字符串的分割来解析各个关键字段及其值。

按照书中所讲，TCP/IP协议是网络的基础，它分为4层，HTTP工作在位于最上层的应用层。以一个典型的客户端/服务器端通信为例，如下图:

一般基于HTTP通信的双方，都是一来一回，即一方（客户端）发起请求（一来），另一方（服务器）响应这个请求（一回）。最常见的通信双方就是我们电脑上常用的浏览器与web服务器，服务器也可以充当客户端向另一个服务器发起HTTP请求。

下面讲讲几个比较具代表性的现象/问题/特点：

一、从浏览器输入域名网址到看到页面都发生了什么

1.浏览器向DNS服务器询问域名对应的IP地址

2.得到域名对应的IP地址后，组装HTTP请求报文

3.由TCP协议将请求报文分割成多个报文段，可靠的传给对方

4.由IP协议搜索对方地址，并传送报文，此过程可能经过多个路由器

5.服务器端的TCP从对方接收到报文

6.服务器端的HTTP解析请求报文，处理并生成响应报文

7.按照相反的顺序，将响应报文发回给浏览器

8.浏览器根据响应报文解析数据（一般为html文档），并渲染页面，文档中也很可能引用了其他资源，这些资源同样使用HTTP协议想服务器请求

二、HTTP是无状态的

为了使服务器能处理更多的请求，在对一个请求作出响应后，服务器就不再保存这个请求有关的任何痕迹。但随着web的不断发展，这种特性日渐不能满足业务需求，典型代表就是购物网站，服务器需要记住用户身份。为了实现保持状态功能，cookie被引入到了web中来。

三、cookie并不是HTTP协议中的标准

cookie虽然不是标准，但已得到广泛应用。在用户认证成功后，服务器会在响应报文中添加唯一的cookie信息，同时在本地保存一份。以后每次这个客户端再次发起请求，会自动带上这个cookie信息，服务器检查这个cookie与本地存储的是否一致，若一致则认为这个用户是成功认证过的，从而实现了保持状态的功能。cookie还可以设置有效期，浏览器将根据响应报文自动处理。

四.请求头与响应头

HTTP协议的请求和响应报文中必定包含HTTP首部。首部内容给客户端和服务器提供了处理请求/响应所必须的信息。

首部信息，涉及到HTTP版本，host名，方法名，缓存策略，连接类型，文本编码，语言，代理，内容类型等众多方面的信息。细节比较多，这里就不赘述了。

五.HTTP请求的几种method方法

  1. OPTIONS：询问服务器这个资源所支持的所有HTTP方法
  2. GET：向指定的资源发出「显示」请求
  3. POST：向指定资源提交数据，请求服务器进行处理（例如提交表单或者上传文件）
  4. PUT：向指定资源位置上传其最新内容
  5. DELETE：请求服务器删除资源
  6. TRACE：回显服务器收到的请求，主要用于测试或诊断
  7. CONNECT：HTTP/1.1协议中预留给能够将连接改为管道方式的代理服务器
  8. PATCH：用于将局部修改应用到资源
    HTTP服务器至少应该实现GET和HEAD方法，其他方法都是可选的。

以上摘自维基百科，这些都只是“约定”，目前市面上大部分web服务器软件都实现了GET和POST方法，其他方法可能由于安全等原因没有被广泛实现。随着web应用越来越复杂，只使用GET和POST方法难以满足日常开发了，后来诞生了REST(Representational State Transfer 表述性状态转移)这种URI设计风格的架构，建议读一读这篇文章 理解本真的REST架构风格. 在我的理解，简单讲，REST只是一种以资源为中心的URI的设计规范，并不是一种技术，或者可以说是指导思想。比如一个用户模块的URI设计。目前大部分服务器并没有实现PUT,PATCH这些方法，所以比较靠谱的办法是在url后追加_method=PUT这种参数，让服务端（比如Spring MVC 框架就支持）应用程序区分是哪种操作。以用户模块举几个例子：

六、简要罗列一下几大类响应码

HTTP响应报文都会带有明显的响应码和相应短语，来标识相应的总体状态，主要有4大类

1)2XX：成功，表示请求被服务器正确处理

A.200 OK:包含完整实体

B.204 No Content:无实体

C.206 Partial Content:部分内容

 

2)3XX：重定向，指示浏览器需要做出某些特殊操作才能正确处理请求

A.301 Moved Permanently 永久重定向

B.302 Found 临时重定向 ，用于缓存

C.303 See Other 应使用GET方法获取资源

D.304 Not Modified 表明请求中的附带条件不合要求

E.307 Temporary Redirect 临时重定向，与302类似，但302精致POST变成GET

 

3）4XX 表明客户端发生错误

A.400 Bad Request 请求报文中存在语法错误

B.401 Unauthorized 未认证

C.403 Forbidden 对资源的请求被拒绝

D.404 资源未找到

 

4）5XX 表明服务器发生错误

A.500 Internet Server Error 可能是web应用存在bug或某些临时故障

B.503 Service Unavailable 表明服务器处于超负荷或停机维护状态，无法处理请求

小结

随着web的发展，HTTP在安全方面也不能满足人们的需求了，于是出现了更安全的HTTPS协议。2014年底，HTML5的定稿，HTTP协议也被扩展开来，引入了更多功能，如Web Socket，WebDAV等。还有Google也为HTTP的发展贡献了自己的力量，HTTP2.0就是基于Google之前推广的SPDY协议设计的。

然而协议终究是协议，只是一种约定或者说是规范，只有实现了HTTP协议的软件（浏览器、服务器）才能在web上良好的工作。

移动页面中的三个apple相关meta标签

今天得知Qcon北京2015刚刚结束，去查看在线PPT的时候，注意到之前就看过好多次的这几个meta标签，研究之！

在ios8的Safari中测试以下meta标签

1.<meta name=”apple-mobile-web-app-capable” content=”yes”>

官网解释：

If content is set to yes, the web application runs in full-screen mode; otherwise, it does not. The default behavior is to use Safari to display web content.

You can determine whether a webpage is displayed in full-screen mode using the window.navigator.standalone read-only Boolean JavaScript property.
意思是设置web应用是否运行在全屏模式

自己做了个demo，无论设置为yes或no，似乎都没什么变化，打印window.navigator.standalone，一直是false。。

 

2.<meta name=”apple-mobile-web-app-status-bar-style” content=”black-translucent”>

官网解释：

This meta tag has no effect unless you first specify full-screen mode as described in apple-[apple-mobile-web-app-capable](https://developer.apple.com/library/safari/documentation/AppleApplications/Reference/SafariHTMLRef/Articles/MetaTags.html#//apple_ref/doc/uid/TP40008193-SW3).

If content is set to default, the status bar appears normal. If set to black, the status bar has a black background. If set to black-translucent, the status bar is black and translucent. If set to default or black, the web content is displayed below the status bar. If set to black-translucent, the web content is displayed on the entire screen, partially obscured by the status bar. The default value is default.
意思是：在全屏模式下，可设置状态栏的颜色，可以为default,black,black-translucent三种值

继续在上面的demo里加入这个设置，无论哪个值，又似乎没什么变化
3.<meta name=”format-detection” content=”telephone=no”>

官网解释：

By default, Safari on iOS detects any string formatted like a phone number and makes it a link that calls the number. Specifying telephone=no disables this feature.
意思是：可设置文本的一些格式化,比如电话号码自动识别，邮件自动识别。默认是开启识别功能的。

经测试，邮件似乎无法识别，电话号码可以，但只有【初始】文档中的一些数字，才可被自动识别为电话号码，通过js后期插入到DOM中的数字是无法识别的，通过weinre调试得知，Safari自动为电话号码包裹上了a标签，并设置了href=”tel:{电话号码}”。请看图：

当设置content=”telephone=no”，自动识别将关闭。

 

不知是不是我测试方法不对呢，还是随着iOS或Safari的版本升级，导致部分功能失效。

又一个CSS3小试——webpack LOGO

昨天晚上准备研究webpack，进入官网猛的吸引我眼球的是它的LOGO，外部的正方体内部嵌一个小立方体，定时旋转，很酷有木有！审查元素一看，纯css3实现，无半个图片像素，好屌的样子，研究之！

先上个图：

先把这2个正方体cube从html里扒下来，拷到我自己的demo里，然后再把相关css也扒下来，发现它的css有310+行，够多的~在本地跑起来，嗯没问题。

 

这些css瞄下来，用了很多css3属性，IE到10才部分支持。然后参考官方版，一点一点的“临摹”，有一些细节不大一样，终于实现了大约90%的效果，去掉了浏览器前缀，只剩下-webkit-animation,因为标准的animation似乎chrome都不支持。最后的代码数是114行。

发布在这里 ： http://runjs.cn/detail/lph1xtgf

CSS3主要使用了

  1. transform-style: preserve-3d //关键！使子元素能够以3d模式呈现
  2. transform-origin //设置旋转轴
  3. rotateX，rotateY，rotateZ //设置正方体的各个面的位置，以及animation设置整个正方体旋转的角度
  4. -webkit-animation && @keyframes //设置动画
  5. radial-gradient //设置背景径向渐变
  6. :after伪类 //协助设置正方体背景
    还有一个backface-visibility，我切换过后，似乎没有什么效果，google之，是用来设置3d模式下在背向屏幕外侧时，是否需要隐藏自身。另外做了一个demo，也就是上面那个按钮。

使用强大的Chrome开发者工具——网络请求篇

距本系列上一篇，也有20天了。这次就拿百度首页开刀吧~建议登陆百度账号，可以看到更多网络请求类型。

网络请求的查看，主要就是在Network这个面板上了，在Elements面板的隔壁。大致上是这样一个界面

1.第一个红色圆圈按钮，用来切换是否记录网络请求，刚打开开发者工具的时候应该是开启（红色）状态，页面上只要发生了网络请求都会在这里产生记录。点击切换为灰色，则接下来的网络请求将不会记录。

2.第二个灰色禁止按钮，用于清空网络请求记录。

3.第三个漏斗按钮，用于开启过滤条件组，蓝色为开启状态。过滤条件组也即紧接着的下面那一行，

第一个文本框用来，输入关键字过滤网络请求记录，实时搜索奥~

后面的一排按钮都是互斥的,All是不做任何过滤，其他的分别用于过滤：文档请求(Documents)，css样式文件(Stylesheet)，图片(Images),audio、video等媒体(Medias),脚本(Scripts),XMLHTTPRequest请求(XHR),字体(Fonts),WebSocket请求(WebSockets),Flash等其他请求(Other)。

最后还有一个Hide data URLs的checkbox，用于切换是否隐藏dataurl，即图片的dataurl。

4.列表图标的按钮，用于切换表格记录中的行视图，大视图或小视图，大视图能够显示更详细的信息。

5.时间轴图标的按钮，当你只关心各个资源的加载时间轴时，它会派上用场。

6.Preserve Log checkbox，若为选中状态则在刷新页面时，不清除旧的网络请求记录，反之清除。

7.Disable Cache checkbox，若为选中状态，则在打开开发者工具的情况下刷新页面，所有静态资源都强制从服务器冲下加载。反之则会根据具体情况使用缓存。

8.最后来看一下记录列表，每列依次是

Name:请求名，

Method:方法(get,post等)，

Status:响应状态，

Type：请求类型，

Initiator：初始化器(请求发起者)，

Size：资源大小，

Time：请求花费时间，

TimeLine：时间线。

每一列都可以点击列头进行排序。

随便点击一个请求记录 ，右侧会出现该请求的详细HTTP请求与响应数据，分为头部，响应预览，响应，cookies，时间点五类信息。尤其是以JSON为响应数据的XHR请求，查看响应预览(Preview)非常人性化，可以直接查看格式化的数据结构，如下图所示

在Header里面，如果请求参数是JSON格式，那么有类似的功能。

DIY零代码搭建OpenVPN服务

清明后上班，同事买了个GreenVPN服务用来番茄，我也去研究了一番，后来想起我的VPS好像也可以搭建VPN服务，因为之前搭建shadowsocks的时候，隔壁就是OpenVPN菜单。遂又是一番折腾，其实也非常简单。其实搭建这两个服务能如此简单，都要归功于搬瓦工的人性化服务~赞一个！下面进入正题

1.购买VPS

可参见我上一篇《DIY零代码搭建 shadowsocks服务》的前半部分

2.一键安装OpenVPN

那么就顺便拿上面这个页面作例子好了。用Chrome打开这个页面：

一、查看元素盒子概况

我们在这篇文章的标题处，右键->选择【审查元素】，如果你当前标签的开发者工具没有开启的话，此操作将开启它，然后Chrome将焦点定位在这个h2元素上，如图：

我们再将鼠标移到的这个被蓝色高亮的节点上，我们会看到页面中对应的标题会被蓝绿橙三种颜色的背景“包围”了，那么蓝色就是元素自身内容的高宽(不一定是你设置的width属性值，视它的box-sizing而定)，绿色是padding，橙色是margin，如图：

橙色下方的668 x 44px 意思是此元素的内容高宽 + padding + margin + border 的最终宽度。前面的h2#activity.rich_media_title，如果你会使用css选择器的话，应该很容易理解，即此元素的id是activity，拥有rich_media_title这个class。

此查看方式可以轻松了解页面上各个元素的和模型概况，之所以是“概况”，因为此时还无法得知内容高宽以及padding和margin的具体值，我们将在CSS篇中介绍如何查看和模型的各个具体数值。

 

二、删除元素节点

这个很简单，直接按delete键即可将高亮的节点删除，包括其子孙节点，当然你也可以在需要删除的节点上右键->选择【Delete】，紧接着页面上相应的节点也消失了，如图：

 

当然，这个删除操作只是临时性的，刷新页面即可恢复。

三、修改元素节点

这个操作，可分为属性修改和节点“outerHTML”的修改。前者其实是可以通过后者实现的。

a.属性修改

我们刷新刚才的页面，把文章标题恢复。依然在标题上右键->选择【审查元素】，在该元素的class属性上双击左键（或右键->选择【Edit Attribute】），接着该节点的class属性便进入编辑状态，我们把rich_media_title这个class改为rich_media_title11111，甚至删掉，回车，页面上的标题瞬间就变小了，其实是应用在rich_media_title这个class上的css属性消失了，也即css选择器选不到这个节点了，那么相应的css设置便失效了~

另外，我们双击该节点里的文本节点，也立即进入编辑状态，修改完后回车，页面上就能看到效果，如图：

作为web开发的你，对outerHTML应该不陌生。此功能相当于在脚本以外，借助浏览器的力量去修改节点的outerHTML。我们再在该标题节点上右键->选择【Edit as HTML】，可以看到整个节点的内容变成了编辑状态。你可以随意修改其中的内容，完毕后，在编辑框的外部点击一下（即blur一下），即可生效，如图：

在Elements的面板中，继续向下浏览，找到某个P标签，最好是比较下面的未在页面窗口中显示的，然后在该节点上右键->选择【Scroll into View】，我们可以看到,此节点内容出现在了页面窗口中。
