介绍两种移动设备调试方案

移动互联网发展势头强劲，掌握移动端的调试非常有必要，我们这次来谈谈移动端的调试。

大约2013年，在一次分享会上，了解到OS X下，Safari可以远程调试iOS设备的网页，只是在那之后很少涉及移动端WEB开发，也就没怎么关注。

2个月前，入职新公司，为移动端WEB开发做准备，尝试了WEINRE的使用。

今天，再次pick起这个话题，google之。大体上有以下几种方案：

  1. OS X Safari(>=6.0)远程调试ios设备中safari中的网页
  2. OS X/Windows Chrome 远程调试 android设备中chrome网页以及webview中的网页
  3. weinre跨平台调试
  4. Adobe Shadow 跨平台测试(基于weinre封装，更加便捷，更加强大)
    方案1和方案3是我实践过的，方案2由于手头没有合适的android设备，没有成功，方案4需要安装Adobe的客户端，一晚上没装成功。

所以，本次暂时只介绍方案1和方案3，另外两种，等实践过再补上。

一、OS X Safari远程调试iOS设备中safari中的网页

此方案，只能用于调试iOS平台Safari网页，并且要求OS X版本>=10.8,Safari版本>=6.0，这两个要求很容易达成。下面是步骤：

5.在OS X中的Safari菜单->”开发”->选择你的iOS设备，将自动列出其打开的网页，选择google那一项，如图：

聊一聊WEB开发阶段ajax的模拟与联调

昨天一个前端同事问我，MAC如何连接到局域网的共享文件夹，我思索了一下，想起来Finder里有个菜单选项”前往”->”连接服务器”，在弹出的对话框中，输入网络地址，比如smb://192.168.xxx.xxx，然后回车即可（顺带提一下，WINDOWS的话，在任一文件夹地址栏或WIN+R，输入 \192.168.xxx.xxx，回车即可）当然这不是本文的重点。后来了解到，是某后端同事要求刚才那位前端同事以这种方式和他联调ajax，号称可以“立即看到效果”，当时我就震惊了o(╯□╰)o，遂决定聊一聊这个话题。
早期，互联网站点基本都是“瘦客户端”，即无异步请求，设计师（或者叫美工?）把psd变成网页后（或添加一些简单的脚本效果），交给“WEB开发攻城狮”填充数据，Done。然而，随着互联网的普及，用户体验的要求越来越高，各种异步请求被引入，后来出现了“胖客户端”甚至单页应用。起初，javascript仅用于表单校验或简单的动画效果，后来为了提升用户体验，javascript的代码量日益增长，尤其是异步请求的增多，WEB开发攻城狮的负担越来越重。

后来，为了提升用户体验，越来越多的公司把WEB开发中的前端开发工作独立出来交给“前端攻城狮”(早在2001年，雅虎就诞生了世界上第一个前端工程师岗位)，让后端同学能够更加专注业务。而不像以前，从前端到数据库一个人统统搞定。那么问题来了，由于不是一个人完成某个功能，就涉及到联调（或者叫做数据对接），尤其是ajax。终于说到主题了囧。在开始开发前，通常前后端会约定好ajax请求的往返数据结构，然后各自朝着这个终点行进。通常有这几种方式：

一.ajax访问静态文本模拟响应

这是最简单的方式，以jquery为例：

如果你是以file协议打开静态页面，在chrome下是被禁止的，需要为启动程序添加参数，windows参考 这个帖子 。推荐使用webstorm这个IDE,打开html默认以访问服务器的方式打开，就没有这个问题了，或者windows自带的IIS服务器。
优点：

简单，不依赖后端

缺点：

  1. 这种方式需要写一个指向本地的url，也就是说，当你模拟ajax完成后，最后提交代码时，仍然需要修改url地址指向正式的地址
  2. 自己模拟的数据与最终后端提供的数据难免有出入，集成进后端时比较可能需要继续修改代码。

二.本地页面直接访问后端服务

等待后端提供好ajax接口，启动web服务，本地打开页面，发起ajax请求，拿到比较真实的数据。当然，这会产生跨域问题，可以利用html5的Allow-Control-Allow-Origin: * ，来允许跨域。如果这个web服务是启动在后端同学的电脑上，那么你就非常可能受他影响了，一旦他的服务挂了，你就无法请求数据了。如果有一台公用服务器来启动这个服务，情况会好些。

优点：

能够拿到比较真实的响应数据，自己调试好后，基本就不用改代码了。

缺点：

  1. 需要额外处理跨域问题，或者可以使用jsonp来避免跨域问题
  2. 依赖后端的进度，受服务器影响大
  3. 开发时的请求地址与最终的请求地址还是会有些许不一致，开发时的ajax地址肯定需要以协议+ip/域名为前缀

三.本地启动项目web服务

等待后端开发好ajax接口，更新代码，在自己电脑上启动项目的服务，比如tomcat，jetty，甚至PHP的wamp。访问本地服务的页面，发起ajax请求。

优点：

ajax地址即为最终的地址，取到的数据结构也是靠谱的，调试通过后，也基本不用改。

缺点：

  1. 依赖后端的进度
  2. 需要在本地启动项目web服务，可能会比较占用资源，对机器性能有要求

四.ftp连接公用服务器

等待后端开发好ajax接口，在公用服务器更新代码，部署服务并启动。使用ftp工具，将修改后的代码上传至服务器，然后访问该服务器上的服务即可看到效果。webstorm提供了remote host功能，经过适当的配置，即可映射本地代码到服务器代码，当修改完本地代码并保存，将自动上传该代码到服务器，然后回到浏览器刷新即可看到修改效果。我还见过这么干的：本地修改完代码，提交，ssh更新服务器代码，浏览器刷新。效率太过地下，就不单独列出来了。

优点:

  1. 也能取得靠谱的数据，ajax地址也是最终的
  2. 由于修改的是公用服务器上的代码，可访问到这个服务的人都能立即看到效果。

  3. 相对于上一种，不用在自己机器上起服务了，节省了机器的内存与cpu
    缺点：

  4. 依赖后端的进度

  5. 有的公司前后端共用这台服务器，如果后端由于某种需要（特别是java），重启服务起，那么你将无法继续调试ajax

五.本地启动轻量级服务，动态分析ajax请求并响应

这也是我目前使用的方式，尤其是单页应用，会产生大量ajax，更加适合这种场景。

通过NodeJS平台的Grunt构建工具，启动node服务（我用的是grunt-connect），拦截ajax请求，动态分析url路径，返回不同的模拟响应。ajax地址可以任意指定，只要做好相应结果的映射或动态分析。

优点：

  1. 不依赖后端，甚至不依赖局域网，本机即可搞定
  2. 如果逻辑正确，调试通过的代码将不用做任何修改，包括ajax地址
  3. 轻量级web服务，不会占用过多资源

  4. 拦截ajax，动态分析，灵活度很高
    缺点：

  5. 由于是模拟响应，与最终后端提供的响应可能有些许出入，数据交互逻辑上，后期集成进后端时可能需要修改

  6. 需要掌握Grunt等工具的用法，加大了复杂度
     

 

总结

以上的模拟/联调方式，各有优缺点，也可以结合起来使用，以提高工作效率。

ps:开篇提到的那种方式，个人不建议使用，因为连版本控制都无法使用了，代码痕迹都无从保存，而且要求网络稳定，直接改别人电脑上的文件，总感觉不妥。

  • 单击左键，立即放开（click） -> 手指点击（tap）
  • 按住左键，不放开（mousedown） -> 手指触碰屏幕，不离开屏幕（press）

    遗憾的是，这里无法模拟多点触碰，因为电脑上只有一个鼠标。鼠标右键不会模拟手势，响应的只是浏览器自身系统级的事件。

Profiles面板

这个面板可以分析页面中的：

2.JS和DOM内存消耗分析(Take Heap Snapshot)

选择Take Heap Snapshot，再点击下面的Take Snapshot按钮，大概一两秒，chrome就为我们生成此时页面中js的各个堆内存情况快照。

3.自定义时间段JS内存分析（Record Heap Allocations）

这个跟第二个功能类似，只不过是提供了起止时间，可以查看一段时间内的内存分配

 

在一般的web开发中，这个面板的功能可能用得很少，因为不涉及大量的脚本程序运算，但在HTML5游戏开发、webGL以及Web APP等需要大量脚本运算的页面，这个功能就变得非常重要，通过这个功能，可以分析出页面帧率为啥提不上去，分析出哪个方法消耗性能太多。

今天就来重新复习一下LESS。

一、LESS是什么

LESS是一种 动态 样式语言，它的任务是将符合LESS语法的代码编译成CSS文件，简化手写传统CSS，提高样式开发的效率。

LESS 将 CSS 赋予了动态语言的特性，如 变量， 继承， 运算， 函数。LESS 既可以在 客户端 上运行 (支持IE 6+, Webkit, Firefox，动态编译LESS文件)，也可以借助Node.js或者Rhino在服务端运行（输出CSS文件，页面直接引用输出的CSS文件）。

对比一下两种方式：

客户端方式：

优点：简单

缺点：LESS编译库会读取本script标签前的所有less文件，并发起ajax读取less内容，动态地将其编译并插入到文档head中，所以当这个过程比较慢的时候（比如less内容很大，ajax响应慢等），页面会出现“无样式”的一个短暂画面，使得用户体验变得糟糕。因为在ajax发出后，文档将继续往下解析，解析文档的速度比ajax响应快的多，在ajax响应之前，被解析的文档内容都是安装浏览器原生样式显示的。

服务端方式：

优点：不会出现客户端方式的“无样式”短暂画面，用户体验好，因为是直接引用css文件。

缺点：需要额外执行编译脚本（在不引入自动化脚本的情况下，需要反复编译less文件，比较繁琐），对开发者提出了新的要求。对此，业界有一种文件监听的策略来减轻开发者的负担，即监听less文件的变动，一旦less文件有变动立即自动编译less文件生成新的css文件。比较常用的node模块有【grunt的watch插件配合grunt的less插件】，与【gulp的watch插件配合gulp的less插件】。

业界主要有两种解决方案：

一、通过useragent区分搜索爬虫和浏览器

比如国内知名AngularJS站点 http://angularjs.cn/ ，根据http头中的useragent首部识别是搜索引擎还是浏览器访问站点。如果是搜索引擎则调用一个专门的中间件生成完整的html文档返回，让爬虫识别；如果是浏览器访问，则返回正常的只有基本结构的页面，后面的解析交给javascript。

不过这种方式，似乎并不保险，据说有些爬虫是匿名，通过useragent判断不出来，还有可能被认为作弊或是“SEO黑帽”。不过经本人浅测试，google和百度都是能够从 http://angularjs.cn/ 正确搜索出结果的。

选择Mac/OS X的九大理由

使用macbook已快两年半，仅以此文为本系列开个头。

首先推荐一个权威人物：池大大，或叫卖桃君。两年多来，我一直在读他的文章，不仅仅是mac的一些知识，还有许多人文科学，受益良多。

本文将会以用户数众多的window系统作为比较，linux等系统本人不甚熟悉，酌情讨论。

正如标题所写，本文是为 OS X系统入门所撰写。对于【把mac预装的OS X系统干掉，而安装windows】的用户，你可以洗洗睡了。

  1. macbook(包括Pro,Air,或是最新的macbook)本身就是很酷的产品，科技感，便携性，不用多说。
  2. mac不需要每天关机，盖上盖子就可以起身，可谓潇洒走一回了。OS X系统具备完善的内存管理，非激活程序(即非当前焦点程序)的内存将被压缩处理，一旦被唤醒为激活程序，其内存会迅速被调出，几乎毫无痕迹。用户几乎感知不到这些切换，更大效率的使用了宝贵的内存。而windows的内存管理广为诟病，一宿不关机就得卡个半天才能反应过来。
  3. 超大的触摸板，以及灵敏的触控体验秒杀其他的notebook。最常用的就是，纵向滚动页面，无缝与手势同步。其次就是横向滚动，与纵向滚动一样轻松。无论纵向，横向还是两者同时，几乎可与手机上的滚动操作相媲美。再有，就是更加丰富高级的多手势操作，比如三指，四指，多方向滑动，pinch等操作，还附带丰富的动画效果。而windows的硬件，一般不具备如此良好的体验，带触屏的win8可能好些，未使用过，不详。
  4. OS X操作系统的稳定性。一般情况下，都不会出什么问题，我到现在为止还没重新装过系统，即使非常偶尔的死机了重新开机还是能够自动恢复至崩溃前的状态，而且系统升级是简单至极。windows的话，尤其做软件开发的，用个半年可能就要重装一次系统，太受伤了。在系统安全性方面，也比windows好上很多，根本不需要安装杀毒软件，windows稍有不慎就会中毒，杀毒软件也是防不胜防。不过客观讲，也可能跟OS X用户数少有关，黑客对用户数多的系统下手成功率要大的多。
  5. windows的软件市场太过混乱，就比如说下个迅雷，搜索到某软件中心可以下载迅雷安装包，谁知下过来的居然是一个xx软件助手，然后再通过这个助手下载真正的迅雷安装包。然后这个助手就默认开机启动了，导致系统内存浪费，忍无可忍卸载它还要花上好几分钟。这种现象太严重了，而在OS X的世界，就清净纯洁的多了。
  6. 正如卖桃君所言，OS X是GUI与命令行的完美结合体，尤其对于软件开发者来讲(windows开发者除外，据说OS X也可能运行.net环境了)。世界上最前沿的技术绝大部分是从UNIX系统率先诞生出来的（应该与它开源有关），而OS X本身就是基于UNIX的，所以命令是兼容UNIX的，那么OS X用户在享受到赏心悦目的GUI程序的同时，也能率先体验全世界开发者的软件分享。另外再说说WEB服务器，绝大多数WEB服务器是LINUX内核，通过OS X可以完美管理LINUX服务器，后期WEB后端开发者，更有理由使用OS X。从某种程度上讲，OS X就是为开发者量身定做的，首先它预装svn，git，python，ruby，Objective-C，swift自是不用说，安装java后，可开发Android和WEB服务端应用，C/C++也是不在话下。最后回到WEB前端，尤其这些年随着NodeJS的崛起，前端攻城狮也不再只是“前端”工程师了，我们可以是“全栈攻城狮”，总之OS X可谓全能开发环境。
  7. 在OS X安装软件（大部分）真的是简单的令人发指(本词也是借用卖桃君- -)。只需一个dmg，打开，把程序拖到Application目录，完毕。不过这样的方式还是少不了手动获取安装包的步骤，在OS X里可以用命令行装软件，只要啪啪敲几个单词，然后等待一小会儿，软件便自行安装完毕，具体下篇文章会讲到。
  8. 从设计行业来说，OS X都是具有其优势的。苹果公司本身就是极具设计灵魂的公司，OS X平台在很久以前就有很多优秀的设计软件，用于处理图形，音频，视频等多媒体。
  9. OS X下拥有漂亮的字体，我尤其喜欢代码的monaco字体
    简单粗暴地讲，如果你是一个设计者或软件开发者（win平台除外），使用OS X绝对不会错。而其他行业从业者，也不会不合适，或者说，OS X是一个出色的办公本。如果你想要一台游戏本，建议还是暂时回到windows的世界吧，也不是说OS X不能玩游戏，是因为国内游戏开发商基本没考虑这个平台，未来就不好说了呵呵。

聊一聊HTTP协议

这些天，开始全面进入项目开发了，卡在一个日历上，无奈自己写一个，回家还继续搞，所以就没写新文章。今天上班状态不错，把日历搞定了，还另外写了一个confirm组件，心情不错。

今天来聊聊HTTP协议。

长久以来对HTTP的认知只停留在GET,POST上，甚至对“协议”二字也模糊不清。直到这一两年从事前端开发，才对HTTP有了稍稍深入的了解。前几个礼拜看完了《图解HTTP》(日·上野宣 著)，这才对工作在web底层的HTTP协议有了一个较全面的认知。

先来讲讲“协议”吧，其实跟我们现实宏观世界类似，就是通信的双方遵循某种约定来一起完成某件事情。而HTTP(Hypertext Transfer Protocol )直译为超“文本”传输协议，本质上还是“文本”，即通讯时传输的只是字符串，通讯双方通过某些特殊字符串的分割来解析各个关键字段及其值。

按照书中所讲，TCP/IP协议是网络的基础，它分为4层，HTTP工作在位于最上层的应用层。以一个典型的客户端/服务器端通信为例，如下图:

一般基于HTTP通信的双方，都是一来一回，即一方（客户端）发起请求（一来），另一方（服务器）响应这个请求（一回）。最常见的通信双方就是我们电脑上常用的浏览器与web服务器，服务器也可以充当客户端向另一个服务器发起HTTP请求。

下面讲讲几个比较具代表性的现象/问题/特点：

一、从浏览器输入域名网址到看到页面都发生了什么

1.浏览器向DNS服务器询问域名对应的IP地址

2.得到域名对应的IP地址后，组装HTTP请求报文

3.由TCP协议将请求报文分割成多个报文段，可靠的传给对方

4.由IP协议搜索对方地址，并传送报文，此过程可能经过多个路由器

5.服务器端的TCP从对方接收到报文

6.服务器端的HTTP解析请求报文，处理并生成响应报文

7.按照相反的顺序，将响应报文发回给浏览器

8.浏览器根据响应报文解析数据（一般为html文档），并渲染页面，文档中也很可能引用了其他资源，这些资源同样使用HTTP协议想服务器请求

二、HTTP是无状态的

为了使服务器能处理更多的请求，在对一个请求作出响应后，服务器就不再保存这个请求有关的任何痕迹。但随着web的不断发展，这种特性日渐不能满足业务需求，典型代表就是购物网站，服务器需要记住用户身份。为了实现保持状态功能，cookie被引入到了web中来。

三、cookie并不是HTTP协议中的标准

cookie虽然不是标准，但已得到广泛应用。在用户认证成功后，服务器会在响应报文中添加唯一的cookie信息，同时在本地保存一份。以后每次这个客户端再次发起请求，会自动带上这个cookie信息，服务器检查这个cookie与本地存储的是否一致，若一致则认为这个用户是成功认证过的，从而实现了保持状态的功能。cookie还可以设置有效期，浏览器将根据响应报文自动处理。

四.请求头与响应头

HTTP协议的请求和响应报文中必定包含HTTP首部。首部内容给客户端和服务器提供了处理请求/响应所必须的信息。

首部信息，涉及到HTTP版本，host名，方法名，缓存策略，连接类型，文本编码，语言，代理，内容类型等众多方面的信息。细节比较多，这里就不赘述了。

五.HTTP请求的几种method方法

  1. OPTIONS：询问服务器这个资源所支持的所有HTTP方法
  2. GET：向指定的资源发出「显示」请求
  3. POST：向指定资源提交数据，请求服务器进行处理（例如提交表单或者上传文件）
  4. PUT：向指定资源位置上传其最新内容
  5. DELETE：请求服务器删除资源
  6. TRACE：回显服务器收到的请求，主要用于测试或诊断
  7. CONNECT：HTTP/1.1协议中预留给能够将连接改为管道方式的代理服务器
  8. PATCH：用于将局部修改应用到资源
    HTTP服务器至少应该实现GET和HEAD方法，其他方法都是可选的。

以上摘自维基百科，这些都只是“约定”，目前市面上大部分web服务器软件都实现了GET和POST方法，其他方法可能由于安全等原因没有被广泛实现。随着web应用越来越复杂，只使用GET和POST方法难以满足日常开发了，后来诞生了REST(Representational State Transfer 表述性状态转移)这种URI设计风格的架构，建议读一读这篇文章 理解本真的REST架构风格. 在我的理解，简单讲，REST只是一种以资源为中心的URI的设计规范，并不是一种技术，或者可以说是指导思想。比如一个用户模块的URI设计。目前大部分服务器并没有实现PUT,PATCH这些方法，所以比较靠谱的办法是在url后追加_method=PUT这种参数，让服务端（比如Spring MVC 框架就支持）应用程序区分是哪种操作。以用户模块举几个例子：

六、简要罗列一下几大类响应码

HTTP响应报文都会带有明显的响应码和相应短语，来标识相应的总体状态，主要有4大类

1)2XX：成功，表示请求被服务器正确处理

A.200 OK:包含完整实体

B.204 No Content:无实体

C.206 Partial Content:部分内容

 

2)3XX：重定向，指示浏览器需要做出某些特殊操作才能正确处理请求

A.301 Moved Permanently 永久重定向

B.302 Found 临时重定向 ，用于缓存

C.303 See Other 应使用GET方法获取资源

D.304 Not Modified 表明请求中的附带条件不合要求

E.307 Temporary Redirect 临时重定向，与302类似，但302精致POST变成GET

 

3）4XX 表明客户端发生错误

A.400 Bad Request 请求报文中存在语法错误

B.401 Unauthorized 未认证

C.403 Forbidden 对资源的请求被拒绝

D.404 资源未找到

 

4）5XX 表明服务器发生错误

A.500 Internet Server Error 可能是web应用存在bug或某些临时故障

B.503 Service Unavailable 表明服务器处于超负荷或停机维护状态，无法处理请求

小结

随着web的发展，HTTP在安全方面也不能满足人们的需求了，于是出现了更安全的HTTPS协议。2014年底，HTML5的定稿，HTTP协议也被扩展开来，引入了更多功能，如Web Socket，WebDAV等。还有Google也为HTTP的发展贡献了自己的力量，HTTP2.0就是基于Google之前推广的SPDY协议设计的。

然而协议终究是协议，只是一种约定或者说是规范，只有实现了HTTP协议的软件（浏览器、服务器）才能在web上良好的工作。
